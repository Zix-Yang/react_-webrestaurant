# 1. 简介

Restaurant Web 是一个面向餐厅场景的 Web 点餐与互动评价系统，采用 React 构建前端界面，结合 Node.js + MySQL 实现后端业务逻辑、接口服务与数据存储。

本系统不仅支持餐厅主页展示、菜单浏览、明星菜品推荐、评分与匿名评论，还进一步扩展了**桌号绑定点餐、拼桌合单、扫码进入网页、用户注册登录、餐厅意见反馈**等功能，使其从单纯的菜品展示平台升级为更贴近真实餐厅业务流程的数字化点餐系统。

---

# 2. 产品概述

## 2.1 产品定位与目标用户

Restaurant Web App 是一款面向餐厅顾客的数字化菜单展示、在线点餐与互动评价平台，其核心定位为：

**“帮助顾客通过扫码进入网页、绑定桌号、浏览菜单、完成点餐并进行评价反馈的一体化餐厅 Web 系统。”**

本产品旨在通过清晰直观的首页展示、结构化的菜单页面、桌号点餐机制、明星菜品推荐、匿名评论功能、餐厅意见反馈模块以及用户账户体系，解决用户在餐厅点餐过程中面临的菜单浏览效率低、热门菜品识别困难、多人拼桌下单不便、反馈入口不足等问题，从而提升餐厅的数字化服务能力与顾客点餐体验。

本产品的主要目标用户包括：

- **首次到店用户**：通过扫码快速进入系统，了解餐厅风格、菜单分类与推荐菜品。
- **堂食顾客**：希望输入桌号后快速浏览菜单并在线点餐。
- **多人用餐用户**：希望实现不同桌号拼桌或合单点餐，提高协同点餐效率。
- **注册用户**：希望保存历史订单、便捷登录并获得更完整的点餐体验。
- **匿名反馈用户**：希望在不登录的情况下依然可以对菜品进行评论与评分。
- **餐厅运营者**：希望通过订单、评分、评论与意见反馈数据优化菜品和服务流程。

---

## 2.2 核心价值主张

本应用的核心价值在于构建一个 **“扫码进入 - 桌号绑定 - 菜单浏览 - 在线点餐 - 用户反馈 - 餐厅优化”** 的闭环体验：

- **扫码即用，降低进入门槛**：顾客可通过扫描餐桌二维码快速进入网页，无需额外下载 App。
- **桌号绑定，提高点餐准确性**：用户进入系统后输入桌号，确保订单准确对应到餐厅中的具体用餐桌位。
- **支持拼桌 / 合并桌号点餐**：在多人聚餐场景下，支持例如“3桌 + 4桌”联合点餐，提升协同下单效率。
- **支持注册登录，增强用户体系**：用户可注册和登录账号，用于管理订单、保存记录或参与更多个性化功能。
- **保留匿名评论，降低反馈门槛**：即使未注册，用户依然可以匿名发表评论，提升参与率。
- **新增餐厅意见栏**：用户可对整体环境、服务体验和上菜效率等提出建议，帮助餐厅获得更全面反馈。
- **突出明星菜品**：根据评分结果展示高评分菜品，帮助顾客快速决策。
- **辅助餐厅运营优化**：订单、评分、评论和意见反馈数据可用于菜品分析、用户偏好识别和服务改进。

# 3. 核心功能模块

## 3.1 餐馆主页展示系统（Restaurant Homepage）

### 3.1.1 布局与交互设计

首页采用 **模块化信息展示布局**，让用户快速完成对餐厅的整体了解，并进入扫码、菜单浏览和点餐流程。

| 元素 | 描述 | 交互细节 |
|------|------|----------|
| 顶部导航栏 | 展示餐厅 Logo、首页入口、菜单入口、点餐入口、评论入口、登录入口 | 支持快速跳转；导航栏可固定 |
| Banner 区域 | 展示餐厅主视觉图、宣传文案和品牌特色 | 用于建立餐厅形象和第一印象 |
| 餐厅信息区 | 展示营业时间、地址、联系方式和整体评分 | 提供基础信息，增强实用性 |
| 推荐内容区 | 展示招牌菜、主推菜或热门类别 | 点击后可进入菜单对应位置 |
| 评论预览区 | 首页展示部分精选评论 | 增强可信度与互动感 |
| 餐厅意见栏 | 用户可对整体用餐体验提出建议 | 用于收集环境、服务、卫生、等待时间等反馈 |
| 点餐入口区 | 提供“输入桌号开始点餐”或“扫码进入点餐”引导 | 强化核心业务转化路径 |
| 页脚区域 | 展示版权信息、联系方式和补充说明 | 增强页面完整性 |

---

### 3.1.2 首页展示逻辑

首页主要承担品牌展示和业务引导作用，其重点不在于完整替代菜单页，而在于帮助用户快速进入点餐流程。

| 功能点 | 描述 | 交互细节与逻辑 |
|--------|------|----------------|
| 品牌展示 | 展示餐厅风格、定位和氛围 | 通过视觉图和简介文案建立认知 |
| 热门推荐 | 展示高关注菜品或高评分菜品 | 点击后进入菜单页或菜品详情 |
| 点餐入口引导 | 明确引导用户输入桌号或扫码点餐 | 在首页设置显著按钮，提升转化效率 |
| 评论预览 | 展示真实评论摘要 | 帮助用户建立信任与参考依据 |
| 意见反馈入口 | 展示“提交餐厅意见”区域 | 用户可快速反馈服务与整体体验问题 |

## 3.4 在线点餐与购物车系统（Online Ordering & Cart）

该模块用于支持用户在网页中完成菜品选择、数量调整和提交订单。

### 3.4.1 点餐功能设计

| 功能点 | 描述 | 交互细节与逻辑 |
|--------|------|----------------|
| 加入订单 | 用户可将菜品加入当前桌号订单 | 点击“加入点餐”按钮即可添加 |
| 数量调整 | 支持增加、减少菜品数量 | 购物车中实时更新 |
| 当前桌号显示 | 点餐页中持续显示当前桌号/拼桌信息 | 避免下错桌 |
| 订单汇总 | 展示已选菜品、数量、价格和总金额 | 支持实时更新 |
| 提交订单 | 用户确认后提交订单 | 订单写入数据库并发送至后台 |

---

### 3.4.2 点餐流程说明

1. 用户扫码进入网页或手动进入首页。  
2. 用户输入桌号，或通过二维码自动绑定桌号。  
3. 用户浏览菜单并将菜品加入订单。  
4. 如有拼桌需求，可追加桌号形成联合订单。  
5. 用户在订单页确认菜品与数量。  
6. 点击“提交订单”后，系统将订单保存到后端。  
7. 餐厅后台可根据桌号接收并处理订单。

## 3.5 用户注册与登录系统（User Registration & Login）

本系统新增完整的前台用户账户功能，用于支持更丰富的个性化服务，但同时保留匿名访问能力。

### 3.5.1 注册登录功能设计

| 功能点 | 描述 | 交互细节与逻辑 |
|--------|------|----------------|
| 用户注册 | 用户通过邮箱/用户名/密码完成注册 | 支持基础表单校验 |
| 用户登录 | 已注册用户登录系统 | 登录后可获取更多个性化功能 |
| 账户状态显示 | 导航栏显示当前登录状态 | 如“欢迎，用户名” |
| 退出登录 | 用户可退出当前账户 | 清除会话信息 |
| 订单记录 | 登录用户可查看历史订单 | 增强账户价值 |

---

### 3.5.2 登录用户与匿名用户并存逻辑

- **登录用户**：可保存订单历史、便捷下单、管理账户信息。  
- **匿名用户**：无需登录也可浏览菜单、输入桌号、完成点餐，并可发布匿名评论。  
- 系统采用“核心业务低门槛 + 账户功能增强型”的设计策略，即**不强制登录，但提供登录带来的附加价值**。

该模块用于收集用户对菜品的满意度评价，并作为明星菜品判断的核心数据来源。

## 3.7 匿名评论系统（Anonymous Comment System）

本系统保留匿名评论机制，以降低反馈门槛并保留真实表达空间。

### 3.7.1 匿名评论功能设计

| 功能点 | 描述 | 交互细节与逻辑 |
|--------|------|----------------|
| 评论输入框 | 用户无需登录即可输入评论内容 | 支持多行文本输入与字数限制 |
| 匿名昵称 | 用户可自定义昵称，也可完全匿名 | 未填写时默认为“匿名用户” |
| 评论发布按钮 | 提交评论内容至系统 | 触发前端校验与后端保存逻辑 |
| 评论列表展示 | 展示菜品历史评论 | 按时间倒序显示 |
| 评论审核 | 管理员审核和管理评论 | 过滤垃圾评论与违规内容 |

---

### 3.7.2 匿名评论与登录评论的兼容策略

- 登录用户可以使用账号身份发表评论。  
- 未登录用户仍可选择匿名发表评论。  
- 为保护低门槛互动体验，**评论功能不强制登录**。  
- 系统可在前端提示：“登录后可保留评论记录，匿名评论也可正常发布。”

# 4. 技术架构与实现

## 4.1 技术栈推荐

| 模块 | 技术栈 | 作用与优势 |
|------|--------|------------|
| 前端框架 | React.js | 提供组件化开发能力，适合构建菜单、购物车、桌号绑定与登录页面 |
| 后端服务 | Node.js + Express | 用于实现用户认证、订单处理、评论管理和 REST API 服务 |
| 数据库 | MySQL | 存储菜品、订单、用户、评分、评论和桌号数据 |
| ORM / 数据访问 | Prisma / Sequelize（可选） | 提升数据库操作的可维护性 |
| 样式方案 | CSS / CSS Modules / Tailwind CSS（可选） | 实现清晰、简洁、商业化界面 |
| 接口通信 | REST API | 用于 React 与 Node.js 后端之间的数据交互 |
| 二维码接入 | QR Code 生成与解析 | 支持用户扫码进入系统并定位桌号 |
| 身份认证 | JWT / Session | 用于用户登录状态管理 |
| 安全校验 | 表单验证 + 内容过滤 + 身份认证 | 提升系统安全性和可维护性 |

---

## 4.2 关键技术难点攻克

| 难点 | 挑战描述 | 解决方案 |
|------|----------|----------|
| 桌号绑定准确性 | 点餐必须准确关联到真实桌号 | 通过二维码预绑定与手动确认双重机制解决 |
| 拼桌合单逻辑 | 多个桌号共同下单时易产生归属混乱 | 后端设计主桌号 + 附加桌号结构 |
| 注册登录与匿名评论并存 | 既要支持账户体系，又要保留匿名反馈 | 通过评论身份分层设计实现 |
| 匿名评分防刷 | 无需登录可能带来重复评分风险 | 结合 Cookie、IP、设备标识做基础限制 |
| 前后端联动 | 菜品、订单、评论、用户状态需实时同步 | 通过 REST API 实现统一数据交互 |
| 意见反馈分类管理 | 反馈内容类型复杂，难以直接统计 | 在后端设计反馈分类字段与管理接口 |

## 4.3 后端架构与数据库设计（Backend Architecture & Database Design）

### 4.3.1 后端系统架构概述

Restaurant Web App 后端采用 **Node.js + Express + MySQL** 方案，负责支撑以下业务：

- 用户注册、登录与身份管理
- 桌号绑定与拼桌订单逻辑
- 菜品管理、评分与评论管理
- 在线点餐与订单处理
- 餐厅意见反馈收集与管理
- 明星菜品计算与展示

# 5. 数据合规与内容安全

## 5.1 数据使用原则

- 对匿名用户仅收集实现评论、评分、反馈和点餐所需的最小数据。  
- 对注册用户仅收集账户功能所必须的信息，如用户名、邮箱和加密密码。  
- 对桌号信息和订单信息进行最小化展示与安全存储。  
- 评论、反馈和订单数据仅用于业务展示与餐厅服务优化。  

---

## 5.2 内容安全与系统安全

- 对评论内容和意见反馈内容进行敏感词过滤和基础审核。  
- 对登录密码进行加密存储。  
- 对订单接口、评分接口、评论接口和反馈接口进行输入校验。  
- 对用户输入内容进行转义，防止 XSS、SQL 注入等常见风险。  
- 管理员后台需具备权限控制与密码加密机制。

# 6. 交互设计要点

- **扫码进入快速**：顾客应能通过扫描二维码迅速进入网页。  
- **桌号状态明确**：用户必须清楚当前订单绑定的桌号与拼桌信息。  
- **菜单浏览高效**：分类、菜品卡片、评分和评论信息应清晰分层。  
- **点餐流程顺畅**：从加菜、改数量到提交订单的流程应尽量简洁。  
- **登录非强制**：核心点餐、评论和反馈功能不应被登录门槛阻断。  
- **意见反馈入口清晰**：首页应设置明显的餐厅意见栏，鼓励用户提出整体建议。  
- **反馈及时明显**：登录成功、绑定桌号成功、评论发布成功、意见提交成功、订单提交成功等状态均需即时提示。  

---

# 7. 设计风格

产品整体风格建议采用 **浅色、简洁明快、商业化且具有食欲导向的视觉设计**。页面应强调菜品图片展示效果，搭配适度留白、统一卡片布局和清晰的排版层级，以提升浏览舒适度与品牌专业度。

在组件设计上，应坚持 **组件抽象化与良好封装性** 的原则，例如将导航栏、桌号绑定模块、扫码引导模块、菜单分类区、明星菜品区域、菜品卡片、购物车、登录注册表单、评分组件、评论列表、评论输入框和餐厅意见栏封装为独立模块，以便后续维护、复用与功能扩展。
