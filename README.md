# 2. 产品概述

## 2.1 产品定位与目标用户

Restaurant Web App 是一款面向餐厅顾客的数字化菜品展示与互动评价平台，其核心定位为 **“帮助用户快速浏览菜单、识别明星菜品并进行匿名反馈的餐馆主页系统”**。

本产品旨在通过清晰直观的首页展示、结构化的菜单页面、菜品评分机制以及匿名评论功能，解决用户在浏览菜品、判断热门推荐和参考他人真实体验时的信息不对称问题，从而提升餐厅的线上展示能力与用户点餐决策效率。

本产品的主要目标用户包括：

- **首次到访用户**：希望快速了解餐厅风格、热门菜品和整体评价。
- **复访用户**：希望快速找到高评分菜品或自己感兴趣的菜单分类。
- **决策犹豫型用户**：希望通过他人评分和评论判断哪道菜更值得尝试。
- **餐厅运营者**：希望通过用户评分与匿名评论收集反馈，优化菜品结构与展示策略。

---

## 2.2 核心价值主张

本应用的核心价值在于构建一个 **“菜品展示 - 用户评分 - 匿名评论 - 明星菜品推荐 - 餐厅优化”** 的闭环体验：

- **高效展示餐厅信息**：通过结构清晰的首页和菜单页面，集中展示餐厅品牌信息、菜品分类和推荐内容，帮助用户快速建立整体认知。
- **突出明星菜品**：菜单页面增加“明星菜品”专区，根据用户评分结果自动展示得分最高的菜品，帮助用户快速识别最受欢迎的选择。
- **降低互动门槛**：用户无需注册或登录即可参与匿名评论，能够更方便地表达真实用餐感受，提升评论参与率。
- **辅助用户点餐决策**：每道菜品均配备评分与评论模块，用户在浏览菜品时可以同步参考其他顾客的真实反馈，降低选择成本。
- **沉淀可运营数据**：评分和匿名评论数据可以帮助餐厅发现高口碑菜品与待优化菜品，从而持续改进菜单与服务。

# 3. 核心功能模块

## 3.1 餐馆主页展示系统（Restaurant Homepage）

### 3.1.1 布局与交互设计

首页采用 **模块化信息展示布局**，旨在让用户在最短时间内完成对餐厅的整体了解，并快速进入菜单浏览与评价流程。

| 元素 | 描述 | 交互细节 |
|------|------|----------|
| 顶部导航栏 | 展示餐厅 Logo、品牌名称、首页入口、菜单入口、评论入口 | 支持点击跳转至首页、菜单页面、评论区域等；导航栏在滚动时可固定 |
| Banner 区域 | 展示餐厅主视觉图、宣传文案和品牌特色 | 用于传达餐厅风格和主打卖点，增强第一印象 |
| 餐厅信息区 | 展示营业时间、地址、联系方式和整体评分概览 | 用户可快速获取基础信息，增强页面实用性 |
| 推荐内容区 | 展示招牌菜、主推菜或热门类别 | 每个推荐项均可点击进入菜单页对应位置 |
| 评论预览区 | 首页展示部分精选匿名评论 | 用于增强可信度和互动感 |
| 页脚区域 | 展示版权信息、联系方式和补充说明 | 增强页面完整性与专业度 |

---

### 3.1.2 首页展示逻辑

首页不是完整菜单的替代，而是餐厅品牌与核心内容的摘要展示页面，其主要目标是引导用户进入更深入的菜单浏览和评分评论流程。

| 功能点 | 描述 | 交互细节与逻辑 |
|--------|------|----------------|
| 品牌展示 | 强调餐厅风格、定位和氛围 | 通过视觉图和简介文案建立品牌印象 |
| 热门推荐 | 快速展示部分高关注菜品 | 点击后可跳转到菜单页面或菜品详情区域 |
| 菜单入口引导 | 引导用户进入完整菜单浏览 | 在首页设置明显入口按钮，提高转化效率 |
| 评论预览 | 展示真实匿名评论摘要 | 让用户在首页即可建立信任感 |

该模块是本产品的核心功能区域，用于展示全部菜单内容，并突出显示评分最高的明星菜品。

“明星菜品”用于突出菜单中最受欢迎、用户评分最高的菜品，帮助新用户快速做出选择。

| 功能点 | 描述 | 交互细节与逻辑 |

| 跳转功能 | 点击明星菜品卡片可跳转至其详情区域 | 缩短用户选择路径，提高页面实用性 |

### 3.2.3 菜品卡片展示逻辑

| 功能点 | 描述 | 交互细节与逻辑 |

| 菜品图片 | 展示菜品视觉效果 | 使用统一尺寸图片，保证页面整体美观 |
| 菜品名称 | 显示菜品标题 | 标题需清晰简洁，便于快速识别 |
| 菜品价格 | 显示当前价格 | 价格信息需醒目，便于用户比较 |
| 平均评分 | 展示该菜品当前平均得分 | 采用星级或数字形式，帮助用户快速判断 |
| 评论数量 | 展示已有评论总数 | 增强评分的可信度和参考价值 |
| 进入评论区 | 引导用户查看该菜品的详细匿名评论 | 可采用按钮、展开面板或跳转形式实现 |

## 3.3 菜品评分系统（Dish Rating System）

该模块用于收集用户对菜品的满意度评价，并作为“明星菜品”判断的核心数据来源。

### 3.3.1 评分功能设计

| 功能点 | 描述 | 技术要求与交互风格 |
|--------|------|--------------------|
| 星级评分 | 用户可对单个菜品进行 1 至 5 星打分 | 点击星星后即时高亮显示当前选择；支持 hover 预览 |
| 平均分计算 | 系统根据所有评分数据计算菜品平均分 | 后端实时更新平均值，并在前端同步展示 |
| 评分成功反馈 | 用户完成评分后给予提示 | 采用提示消息或轻量反馈增强确认感 |
| 排序支持 | 菜品支持按评分高低排序 | 帮助用户快速筛选高分菜品 |
| 明星菜品联动 | 评分结果影响明星菜品展示 | 评分越高的菜品越有机会进入明星专区 |

---

### 3.3.2 评分逻辑说明

由于本产品不强制用户登录，因此评分功能采用轻量化匿名机制，但仍需具备基础控制能力。

- 用户无需注册或登录即可对菜品进行评分。
- 为防止同一用户在短时间内重复刷分，可结合浏览器本地缓存、Cookie、IP 限制等方式做基础控制。
- 平均评分建议保留一位小数，以增强展示统一性。
- 当菜品评分人数较少时，可增加“评价人数较少，仅供参考”的提示。
- 明星菜品的筛选逻辑以用户平均评分为核心标准。

## 3.4 匿名评论系统（Anonymous Comment System）

该模块用于收集用户对菜品的文字反馈，帮助其他顾客更深入地理解菜品特点，也为餐厅优化提供真实参考。

### 3.4.1 匿名评论功能设计

| 功能点 | 描述 | 交互细节与逻辑 |
|--------|------|----------------|
| 评论输入框 | 用户无需登录即可输入评论内容 | 支持多行文本输入，并提供字数限制提示 |
| 匿名昵称 | 用户可选择填写昵称，也可直接匿名发布 | 若未填写，则使用默认名称如“匿名用户” |
| 评论发布按钮 | 提交评论内容至系统 | 点击后触发前端校验与后端保存逻辑 |
| 评论列表展示 | 展示该菜品的全部历史评论 | 按时间倒序或热度排序显示，并附带评分与时间信息 |
| 评论审核 | 管理员可对评论进行审核和管理 | 用于过滤垃圾评论与违规内容 |

---

### 3.4.2 评论内容展示逻辑

为提升匿名评论模块的可读性与参考价值，每条评论建议包含以下字段：

| 展示字段 | 描述 | 备注 |
|----------|------|------|
| 昵称/匿名名称 | 评论发布者名称 | 可默认为“匿名用户” |
| 用户评分 | 与评论绑定的星级评分 | 便于其他用户理解评论背景 |
| 评论正文 | 用户对菜品口味、分量、摆盘等方面的评价 | 为其他顾客提供具体参考 |
| 发布时间 | 评论提交时间 | 帮助用户判断评论时效性 |

---

### 3.4.3 评论审核与内容规范

为了保证评论区质量与页面可维护性，匿名评论系统需具备基础内容治理能力：

- 对敏感词、侮辱性语言和垃圾评论进行过滤。
- 对空评论、纯符号评论和重复评论进行前端校验。
- 管理员可对违规评论执行删除、隐藏或审核操作。
- 匿名评论虽降低了参与门槛，但仍需通过基础限制机制防止恶意刷屏。

## 3.5 后台管理模块（Admin Management）

由于本产品采用匿名评分与匿名评论模式，因此无需为普通用户设计完整的前台登录系统。用户模块可弱化为餐厅运营者使用的后台管理模块。

| 模块 | 描述 | 关键技术与功能 |
|------|------|----------------|
| 菜品管理 | 管理员新增、编辑或删除菜品 | 维护图片、价格、描述和分类信息 |
| 评论管理 | 管理员查看和处理匿名评论 | 删除违规评论、隐藏垃圾内容 |
| 评分管理 | 查看菜品评分统计情况 | 用于分析用户偏好和热门菜品 |
| 明星菜品管理 | 根据评分结果展示明星菜品 | 可自动展示最高分菜品，也可进行后台确认 |

# 4. 技术架构与实现

## 4.1 技术栈推荐

为确保餐馆主页系统具备良好的可维护性、交互流畅性和后续扩展能力，推荐以下技术栈：

| 模块 | 技术栈 | 作用与优势 |
|------|--------|------------|
| 前端框架 | React.js | 提供组件化开发能力，适合构建交互清晰、结构分明的餐馆主页与菜单页面 |
| 后端服务 | PHP + WordPress | 适合快速搭建内容管理型网站，便于实现后台菜品管理、评论管理和数据接口扩展 |
| 数据库 | MySQL | 存储菜品信息、评分数据、匿名评论和分类信息 |
| 样式方案 | CSS / CSS Modules / Tailwind CSS（可选） | 用于实现浅色、简洁、商业风格的页面设计 |
| 接口通信 | REST API | 用于 React 前端与 WordPress / PHP 后端之间的数据交互 |
| 安全校验 | 表单验证 + 内容过滤机制 | 防止垃圾评论、恶意输入和基础安全风险 |

---

## 4.2 关键技术难点攻克

| 难点 | 挑战描述 | 解决方案 |
|------|----------|----------|
| 匿名评分防刷 | 用户无需登录，可能出现重复评分问题 | 通过 Cookie、本地缓存、IP 限制等方式进行基础控制 |
| 明星菜品动态展示 | 明星菜品需依据评分变化动态更新 | 后端定期或实时计算平均分，前端调用接口展示最高分菜品 |
| 匿名评论治理 | 匿名发布降低门槛，也提高垃圾评论风险 | 前后端同时进行内容校验、敏感词过滤和后台审核 |
| 前后端联动 | React 与 WordPress/PHP 之间需要数据同步 | 通过 REST API 获取菜品、评分、评论与明星菜品数据 |

---

## 4.3 后端架构与数据库设计（Backend Architecture & Database Design）

### 4.3.1 后端系统架构概述

Restaurant Web App 后端采用基于 **PHP + WordPress** 的内容管理与接口服务方案，以支持菜单展示、匿名评分、匿名评论和后台管理等核心功能。

核心组件如下：

- **WordPress 后台管理系统**：负责菜品内容维护、评论审核和基础数据管理。
- **PHP 业务逻辑层**：负责处理评分、评论、明星菜品计算等业务逻辑。
- **REST API 接口层**：向 React 前端提供菜品列表、评分数据、评论数据和明星菜品数据。
- **MySQL 数据库**：存储所有结构化业务数据。

---

### 4.3.2 数据库设计（ER / Schema 关键表）

| 表名 | 描述 | 关键字段 | 备注 |
|------|------|----------|------|
| dishes | 菜品信息 | `dish_id (PK)`, `name`, `description`, `price`, `image_url`, `category_id`, `average_rating` | 存储菜品核心展示内容 |
| categories | 菜品分类 | `category_id (PK)`, `category_name` | 用于前菜、主菜、甜品、饮品等分类管理 |
| ratings | 菜品评分记录 | `rating_id (PK)`, `dish_id (FK)`, `score`, `ip_hash`, `created_at` | 用于记录匿名评分信息 |
| comments | 匿名评论记录 | `comment_id (PK)`, `dish_id (FK)`, `nickname`, `content`, `score`, `created_at`, `status` | 存储匿名评论内容及审核状态 |
| admins | 管理员信息 | `admin_id (PK)`, `username`, `password_hash` | 仅供后台管理使用 |

---

### 4.3.3 核心 API 接口定义

| 模块 | 接口/路径 | 方法 | 描述 |
|------|-----------|------|------|
| 菜品 | `/api/dishes` | GET | 获取菜单页菜品列表及分类信息 |
| 菜品 | `/api/dishes/{id}` | GET | 获取单个菜品的详细信息、评分和评论 |
| 明星菜品 | `/api/dishes/featured` | GET | 获取评分最高的明星菜品列表 |
| 评分 | `/api/dishes/{id}/rating` | POST | 提交匿名评分 |
| 评论 | `/api/dishes/{id}/comments` | POST | 发布匿名评论 |
| 评论 | `/api/dishes/{id}/comments` | GET | 获取该菜品的匿名评论列表 |
| 后台管理 | `/api/admin/comments` | GET | 获取评论管理数据 |
| 后台管理 | `/api/admin/dishes` | POST/PUT/DELETE | 新增、修改或删除菜品 |

---

### 4.3.4 关键业务逻辑实现

#### 1. 匿名评分流程

1. 用户进入菜单页面或菜品详情区域。  
2. 用户点击星级组件选择评分。  
3. 前端将评分数据发送至 PHP 后端接口。  
4. 后端根据 IP、Cookie 或本地标识判断是否存在短时间重复评分行为。  
5. 后端保存评分记录，并重新计算该菜品平均评分。  
6. 前端刷新当前菜品评分显示。  

#### 2. 匿名评论发布流程

1. 用户在评论输入框填写评论内容。  
2. 用户可填写昵称，也可直接匿名发布。  
3. 前端对输入内容进行长度与合法性校验。  
4. 用户点击“发布评论”按钮后，请求发送到评论接口。  
5. 后端完成敏感词过滤、内容转义和数据存储。  
6. 评论保存成功后，前端即时刷新评论列表，并展示成功提示。  

#### 3. 明星菜品展示流程

1. 系统根据所有菜品的平均评分进行排序。  
2. 后端筛选出评分最高的菜品作为明星菜品。  
3. React 前端调用明星菜品接口。  
4. 菜单页面在顶部或显著区域展示明星菜品专区。  
5. 用户点击明星菜品卡片后，可跳转至对应菜品详情或评论区域。

# 5. 数据合规与内容安全

## 5.1 数据使用原则

由于本产品采用匿名评论与匿名评分机制，因此无需收集普通用户的完整账户信息，但仍需遵循基本的数据合规原则：

- 仅收集实现评分与评论功能所必需的最小数据。
- 对匿名评分涉及的 IP、Cookie 或设备标识应进行最小化处理。
- 评论内容应明确告知仅用于页面展示与餐厅服务优化。
- 平台应避免直接暴露任何敏感技术字段或后台信息。

---

## 5.2 内容安全与系统安全

- 对匿名评论内容进行基础审核，防止违规内容传播。
- 对评分接口增加基础限制机制，防止恶意刷分。
- 用户输入内容应进行转义和校验，防止 XSS、SQL 注入等常见风险。
- 管理员后台需具备权限控制与密码加密机制，保护系统安全。

# 6. 交互设计要点

- **首页导向清晰**：用户进入网站后，应能快速理解餐厅定位并找到菜单入口。
- **菜单结构清晰**：分类、菜品卡片、评分和评论信息应分层明确，提升浏览效率。
- **明星菜品突出**：菜单页面应在显著位置展示评分最高的明星菜品，并通过视觉高亮增强吸引力。
- **匿名评论门槛低**：让用户无需登录即可快速发表看法，减少操作阻力。
- **反馈及时明显**：评分成功、评论发布成功、输入错误等状态需有即时提示，增强交互流畅性。

# 7. 设计风格

产品整体风格建议采用 **浅色、简洁明快、商业化且具有食欲导向的视觉设计**。页面应强调菜品图片展示效果，搭配适度留白、统一卡片布局和清晰的排版层级，以提升浏览舒适度和品牌专业度。

在组件设计上，应坚持 **组件抽象化与良好封装性** 的原则，例如将导航栏、菜单分类区、明星菜品区域、菜品卡片、评分组件、匿名评论列表和评论输入框封装为独立模块，以便后续维护、复用与功能扩展。
